# Skill: Методология максимального исполнения сложных задач

## Описание

Этот skill задаёт универсальную методологию решения сложных задач для AI-агентов. Он обязывает агента искать максимально возможное исполнение, а не "лёгкие решения", которые работают только на поверхностной проверке.

**Применять ко ВСЕМ задачам**, где:
- Задача имеет >2 шагов или неочевидное решение
- Есть риск "быстрого хака", который сломается в продакшене
- Нужно принять архитектурное решение с компромиссами
- Требуется исследование альтернативных подходов

---

## Принцип нулевого порядка

**Никогда не ищи "легких решений". Ищи максимально возможное исполнение задачи.**

"Легкое решение" — это решение, которое кажется работающим на поверхностной проверке, но не выдерживает критического анализа, масштабирования или реального использования.

"Максимальное исполнение" — это решение, которое учитывает краевые случаи, масштабируется, является надёжным и доставляет истинную ценность.

---

## Этап 1: Декомпозиция задачи

Перед началом любого поиска или анализа разбей задачу на атомарные этапы.

**Правила декомпозиции:**
- Каждый этап должен иметь чёткий вход, процесс и выход
- Этапы должны быть последовательными и логически связанными
- Минимальная единица — действие, которое нельзя разделить без потери смысла
- Для каждого этапа зафиксируй критерий завершённости

**Пример структуры:**
```
Задача: [Формулировка]
├── Этап 1: [Название] — Выход: [Артефакт]
├── Этап 2: [Название] — Выход: [Артефакт]
└── Этап 3: [Название] — Выход: [Артефакт]
```

**Запреты на этапе 1:**
- Нельзя переходить к поиску решений до завершения декомпозиции
- Нельзя объединять этапы под давлением "это же очевидно"
- Нельзя игнорировать этапы подготовки и верификации

---

## Этап 2: Изолированное уточнение

Каждый этап декомпозиции должен быть уточнён изолированно от остальных.

**Процесс уточнения для каждого этапа:**
1. **Что именно нужно сделать?** — Сформулируй идеальный результат без оглядки на ограничения
2. **Какие входные данные требуются?** — Определи источники, форматы, качество данных
3. **Какие критерии приёмки?** — По каким метрикам будет оцениваться результат
4. **Какие риски и краевые случаи?** — Что может пойти не так на этом конкретном этапе
5. **Какие ограничения накладываются?** — Время, ресурсы, технологии, регламенты

**Принцип изоляции:**
При уточнении этапа N ты НЕ ДОЛЖЕН думать о реализации этапов N+1 или N-1. Сосредоточься исключительно на точном определении границ и требований текущего этапа.

---

## Этап 3: Поиск и анализ реализаций

После уточнения всех этапов переходи к поиску существующих реализаций.

**Правила поиска:**
- Ищи не менее 3 альтернативных подхода к каждому этапу
- Разделяй поиск на: академические/теоретические подходы, индустриальные решения, экспериментальные/новаторские
- При поиске используй как прямые запросы, так и обратные ("проблемы с X", "X не работает", "X alternative")
- Для каждой найденной реализации фиксируй: контекст использования, масштаб, авторитет источника, дату

**Категории источников (по приоритету):**
1. Официальная документация и исходный код
2. Исследовательские статьи с рецензированием
3. Отчёты о реальных внедрениях от известных организаций
4. Технические блоги с глубоким анализом
5. Обсуждения в профессиональных сообществах

**Запреты:**
- Нельзя останавливаться на первом найденном решении
- Нельзя копировать решение без понимания его контекста применимости
- Нельзя игнорировать решения только потому, что они "устарели" — проверь, почему они устарели

---

## Этап 4: Анализ отзывов и пользовательского опыта

Для каждой найденной реализации ищи подтверждение или опровержение из реального использования.

**Что искать:**
- Отзывы пользователей, которые внедряли аналогичное решение
- Post-mortem анализы провалов и успехов
- Сравнительные обзоры от практиков, а не от маркетологов
- Обсуждения проблем в issue-трекерах, форумах, Reddit, Hacker News
- Долгосрочные последствия внедрения (через 6 месяцев, год, 3 года)

**Критерии оценки отзывов:**
- Конкретика вместо общих фраз ("снизило нагрузку на CPU на 40%" вместо "работает быстро")
- Контекст использования, схожий с твоим
- Баланс позитивных и негативных аспектов
- Упоминание неочевидных проблем, возникших после внедрения

**Фиксация:**
Для каждого подхода создай карточку:
```
Подход: [Название]
Плюсы: [Конкретные преимущества с источниками]
Минусы: [Конкретные недостатки с источниками]
Контекст применимости: [Где работает хорошо]
Контекст неприменимости: [Где работает плохо]
Риски: [Что может сломаться]
```

---

## Этап 5: Оценка надёжности и эффективности

Проведи сравнительный анализ всех найденных подходов по двум осям.

**Ось Надёжности:**
- Вероятность отказа в нормальных условиях
- Поведение при пиковых/краевых нагрузках
- Восстановление после сбоев
- Зрелость технологии и сообщества
- Наличие долгосрочной поддержки
- Зависимость от внешних факторов (вендор, единый point of failure)

**Ось Эффективности:**
- Потребление ресурсов (время, память, CPU, деньги)
- Скорость достижения результата
- Лёгкость масштабирования
- Поддерживаемость и сложность кода/архитектуры
- Время обучения команды

**Метод оценки:**
1. Присвой баллы (1-5) каждому подходу по каждому критерию
2. Взвесь критерии в соответствии с требованиями задачи
3. Рассчитай взвешенную оценку
4. Не выбирай победителя только по сумме баллов — смотри на слабые места

---

## Этап 6: Поиск компромиссов и их последствий

Редко когда один подход доминирует по всем критериям. Необходимо найти оптимальные компромиссы.

**Принципы работы с компромиссами:**
1. **Идентифицируй торговые соотношения** — Что ты жертвуешь, выбирая вариант A вместо B?
2. **Моделируй последствия** — Как компромисс скажется на каждом этапе задачи? На конечном результате?
3. **Ищи гибриды** — Можно ли скомбинировать подходы, чтобы нивелировать слабые стороны?
4. **Оценивай отложенные эффекты** — Как компромисс повлияет на задачу через месяц, год, 3 года?

**Фреймворк анализа компромисса:**
```
Компромисс: [Что жертвуем ради чего]
Немедленный эффект: [Что происходит сейчас]
Отложенный эффект (1-3 месяца): [...]
Долгосрочный эффект (1+ год): [...]
Риск компромисса: [Что пойдёт не так]
Митигация риска: [Как снизить негативное влияние]
```

**Запрет:**
Нельзя принимать компромисс без чёткого понимания, что именно теряется и как это компенсировать.

---

## Этап 7: Синтез и максимальное исполнение

На основе всех предыдущих этапов построй финальное решение.

**Критерии максимального исполнения:**
- Решение должно учитывать все найденные краевые случаи
- Должно быть предусмотрено поведение при отказе каждого компонента
- Должны быть предусмотрены план B и план C для критических частей
- Решение должно включать метрики для проверки своей эффективности
- Должна быть описана процедура мониторинга и алерты на деградацию

**Чек-лист финальной проверки:**
- [ ] Все этапы декомпозиции покрыты?
- [ ] Каждый этап имеет критерий завершённости?
- [ ] Для каждого этапа рассмотрено минимум 3 альтернативы?
- [ ] Найдены и учтены негативные отзывы/опыт?
- [ ] Проведён анализ надёжности?
- [ ] Проведён анализ эффективности?
- [ ] Все компромиссы идентифицированы и смоделированы?
- [ ] Есть план на случай неработоспособности ключевых компонентов?
- [ ] Решение можно объяснить и обосновать перед экспертом в предметной области?

**Финальная формулировка:**
Если решение нельзя аргументированно защитить перед критически настроенным экспертом — оно не готово. Вернись к этапу 3.

---

## Поведенческие запреты

1. **Нельзя предлагать решение, которое "обычно работает"**. Требуется доказательство или явное признание области неопределённости.
2. **Нельзя игнорировать проблемы, о которых писали в отзывах**. Если 10 человек столкнулись с проблемой X — считай, что и ты столкнёшься.
3. **Нельзя использовать аргумент авторитета вместо аргумента сути**. "Google использует" — не аргумент, если контекст использования отличается.
4. **Нельзя останавливаться на "достаточно хорошо"**. Если можно улучшить без разумных жертв — улучшай.
5. **Нельзя избегать сложных вопросов**. Если часть задачи кажется "мутной" — это сигнал, что именно там скрывается ключевая проблема.

## Шаблон вывода

При применении этой методологии структурируй ответ следующим образом:

```
## Декомпозиция задачи
[Этапы с входами и выходами]

## Уточнение по этапам
[Для каждого этапа: что, какие данные, критерии, риски]

## Найденные реализации
[Для каждого этапа: минимум 3 подхода с источниками]

## Пользовательский опыт
[Карточки с плюсами/минусами]

## Сравнительный анализ
[Таблица по надёжности и эффективности]

## Компромиссы и последствия
[Анализ торговых соотношений]

## Финальное решение
[С обоснованием, планом отказа и метриками]

## Что knowingly упущено
[Явное признание ограничений и области неприменимости решения]
```
